# Git 컨벤션

## 브랜치 네이밍 규칙

### 형식

```
<type>/<간단한_설명>-#<issue_number>
```

### 타입

| Type       | 설명        | 예시                        |
| ---------- | ----------- | --------------------------- |
| `feat`     | 새로운 기능 | `feat/portfolio-crud-#15`   |
| `fix`      | 버그 수정   | `fix/auth-token-expire-#23` |
| `refactor` | 리팩토링    | `refactor/user-service-#8`  |
| `hotfix`   | 긴급 수정   | `hotfix/login-error-#45`    |
| `chore`    | 빌드, 설정  | `chore/docker-setup-#5`     |
| `docs`     | 문서 작업   | `docs/api-spec-#12`         |

### 규칙

- 소문자 사용
- 단어 구분은 하이픈(`-`)
- Issue 번호 필수 포함

## 커밋 메시지 컨벤션

### 형식

```
<type>: <subject> (#<issue_number>)

<body>
```

### 타입

| Type       | 설명                                        |
| ---------- | ------------------------------------------- |
| `feat`     | 새로운 기능 추가                            |
| `fix`      | 버그 수정                                   |
| `docs`     | 문서 수정                                   |
| `style`    | 코드 포맷팅, 세미콜론 누락 등 (기능 변경 X) |
| `refactor` | 코드 리팩토링                               |
| `test`     | 테스트 코드 추가/수정                       |
| `chore`    | 빌드 설정, 패키지 매니저 등                 |
| `rename`   | 파일/폴더 이름 변경 또는 이동               |
| `remove`   | 파일 삭제                                   |

### 커밋 메시지 규칙

1. **제목은 50자 이내**
2. **제목 끝에 마침표 없음**
3. **제목은 명령문으로 작성** (한글: ~구현, ~수정, ~추가)
4. **본문은 72자마다 줄바꿈**
5. **본문에 What, Why 설명**

### 예시

```bash
# 좋은 예시
git commit -m "feat: 포트폴리오 생성 API 구현 (#15)

- POST /portfolios 엔드포인트 추가
- CreatePortfolioDto 정의
- Swagger 문서 업데이트"

# 간단한 커밋
git commit -m "fix: 토큰 만료 시간 오류 수정 (#23)"

# 문서 수정
git commit -m "docs: API 명세서 업데이트 (#12)"
```

### 금지 사항

AI가 생성했다는 표시 **절대 금지**:

```bash
# 절대 사용 금지
Co-Authored-By: Claude <noreply@anthropic.com>
Generated with Claude Code
Generated by AI
```

## 이슈 제목 컨벤션

### 형식

```
[TYPE] 설명
```

### 예시

```
[Feat] 포트폴리오 CRUD API 구현
[Fix] 로그인 토큰 갱신 오류 수정
[Refactor] 사용자 서비스 리팩토링
[Chore] Docker 설정 추가
[Docs] API 문서 업데이트
```

## PR 제목 컨벤션

### 형식

```
TYPE: 설명 (#이슈번호)
```

### 예시

```
Feat: 포트폴리오 CRUD API 구현 (#15)
Fix: 로그인 토큰 갱신 오류 수정 (#23)
Refactor: 사용자 서비스 리팩토링 (#8)
Chore: Docker 설정 추가 (#5)
Docs: API 문서 업데이트 (#12)
```

## 코드 리뷰 체크리스트

PR 생성 전 확인사항:

- [ ] 브랜치명이 컨벤션을 따르는가?
- [ ] 커밋 메시지가 컨벤션을 따르는가?
- [ ] 관련 Issue가 연결되어 있는가?
- [ ] 로컬에서 빌드/테스트가 통과하는가?
- [ ] Docker 빌드가 성공하는가? (Dockerfile 변경 시)
- [ ] 불필요한 console.log가 없는가?
- [ ] 코드 포맷팅이 적용되었는가?

## Docker 빌드 가이드

### pnpm 버전 관리

Dockerfile의 pnpm 버전은 로컬 개발 환경과 일치시켜야 합니다:

```dockerfile
# package.json의 packageManager 필드 확인 후 동일한 버전 사용
RUN corepack enable && corepack prepare pnpm@10.28.0 --activate
```

### npm scripts의 CI 환경 대응

Git hooks 관리 도구(husky 등)는 `.git` 폴더가 없는 CI/Docker 환경에서 실패할 수 있습니다.
`prepare` 스크립트는 `.git` 폴더 존재 여부를 확인해야 합니다:

```json
{
    "scripts": {
        "prepare": "node -e \"if (require('fs').existsSync('.git')) { require('child_process').execSync('husky', {stdio: 'inherit'}) }\""
    }
}
```

**주의사항:**

- `.dockerignore`에 `.git`이 포함되어 있어 Docker 컨테이너에는 Git 정보가 없음
- `pnpm install` 실행 시 `prepare` 스크립트가 자동으로 실행되므로 조건부 처리 필수
- `.git`이 있는 환경에서는 husky 실행 오류가 정상적으로 전파되어 설정 문제를 즉시 발견 가능

### 로컬 Docker 빌드 테스트

PR 생성 전 로컬에서 Docker 빌드를 테스트하세요:

```bash
# 빌드 테스트
docker build -t folioo-server .

# 전체 단계 테스트 (DB 포함)
docker compose up -d
```
